# QA レポート: 架電レコメンドツール開発

## 1. データパイプラインのテストケース

| テスト項目                       | 入力例                              | 期待結果                                                              | 優先度 |
|----------------------------------|------------------------------------|----------------------------------------------------------------------|--------|
| NULL値処理                      | `NULL` (最終購入日がNULL)         | 最終購入日は「不明」に補完されること                                          | 高     |
| 重複データ排除                  | 同一メールアドレスを持つ顧客データ | 重複したメールアドレスの行は削除されること                                       | 中     |
| 文字コード混在処理              | 半角・全角が混在する企業名       | 企業名が適切に正規化されること                                               | 高     |
| 不正な日付形式処理              | `2023-99-99` (不正な日付形式)    | エラーを返し、処理が行われないこと                                         | 中     |
| データソースA取込失敗時の挙動確認 | 不正なCSVファイル                  | エラーメッセージが表示され、適切に処理が中断されること                             | 中     |
| データソースB取込失敗時の挙動確認 | 存在しないExcelファイル            | エラーメッセージが表示されること                                             | 中     |
| データソースC取込失敗時の挙動確認 | 存在しないOCRテキストファイル     | エラーメッセージが表示されること                                             | 中     |

## 2. API テスト仕様

### 正常系・異常系・境界値のテストケース一覧

| テスト項目                       | 入力例                             | 期待結果                                 | 優先度 |
|----------------------------------|-----------------------------------|---------------------------------------|--------|
| 架電優先リスト取得（正常系）      | 有効なリクエスト                  | 正常な架電優先リストが返されること           | 高     |
| 顧客詳細取得（正常系）            | 存在する顧客ID (例: 1)           | 指定した顧客の詳細情報が返されること        | 高     |
| トークスクリプト生成（正常系）     | 有効な顧客ID (例: 1)             | スクリプトヒントが生成されていること        | 高     |
| 架電優先リスト取得（異常系）      | 無効なリクエスト                  | 400エラーが返されること                     | 中     |
| 顧客詳細取得（異常系）            | 存在しない顧客ID (例: 999)       | 404エラーが返されること                     | 中     |
| トークスクリプト生成（異常系）     | 無効な顧客ID (例: -1)            | 404エラーが返されること                     | 中     |
| 残りのテストケース（境界値）      | 有効な顧客IDの上限 (例: 最大値)   | 正常に詳細情報が返されること                | 中     |

### レスポンスタイムの検証基準
- APIの各エンドポイントにおいて、レスポンス時間は2秒以内である必要があること。

## 3. E2E テストシナリオ

### 営業担当者の課題A・B・Cを解決できるかを検証するシナリオ
1. **課題Aテストシナリオ**: 
   - 前回連絡から時間が空き、購入金額が大きい顧客を特定するためのリストが表示されることを確認する。

2. **課題Bテストシナリオ**: 
   - スマートフォンでログインし、顧客の購買履歴・商談内容を確認できることを検証する。

3. **課題Cテストシナリオ**: 
   - AIが生成したトークスクリプトが表示されることを確認する。

### スマートフォンからのアクセスシナリオ
- スマートフォンからシステムにアクセスし、顧客情報の表示・トークスクリプトの表示など、全機能が適切に動作することを確認する。

## 5. 品質リスク一覧

| リスク                                  | 内容                                           | 軽減策                                   |
|-----------------------------------------|------------------------------------------------|------------------------------------------|
| NULL値によるエラー                      | データソースAにNULL値が存在する場合              | データ補完処理を強化する                   |
| 表記揺れによるデータ不整合              | データソースBに全角/半角揺れが存在する場合       | データ正規化処理を実装する                  |
| 未登録データの取り込み                  | データソースCで未構造化データがある場合         | 取り込み前の検証処理を追加する              |
| システム稼働率が95%を下回る場合         | 運用中の障害によるダウンタイムの増加                 | モニタリングシステムを導入し、早期障害検知を行う |
| レスポンスタイムが基準を超過する場合    | 負荷テスト時にパフォーマンスが低下する可能性      | 負荷分散及びキャッシュ戦略を検討する        |
| APIの不備による顧客の業務影響           | 不完全な機能により営業プロセスが停止すること          | ユーザビリティテストを強化し、機能の確認を徹底する |
